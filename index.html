📋 後續追蹤：為什麼會壞？

1. Microsoft 退役 Incoming Webhook

Microsoft 公告將退役舊版 Office 365 連接器（含 Incoming Webhook），要求遷移至 Teams Workflows（Power Automate）。舊版 Webhook URL 將於 2026 年 4 月 30 日完全停止運作。

2. TLS 1.0 / 1.1 全面停止支援

Microsoft 已全面停止支援 TLS 1.0 與 TLS 1.1。舊版伺服器（如舊 .NET Framework）預設使用 TLS 1.0/1.1 時，Teams Webhook 伺服器會拒絕握手。所有 Microsoft 365 服務必須使用 TLS 1.2 或更高。

3. 根憑證遷移：Baltimore → DigiCert

Microsoft 正在將根憑證從 Baltimore CyberTrust Root 遷移到 DigiCert Global Root G2（舊憑證 2025 年過期）。如果伺服器的 Trust Store 太舊、沒有新的 DigiCert 根憑證，就會無法信任 Teams Webhook 的新憑證。

💡 小知識

TLS 握手與 Proxy 的關係

TLS 握手等於「安檢」：傳送 HTTP 資料前，雙方先對齊加密規則。微軟目前不收舊規格，若發送端還在用 TLS 1.0/1.1，握手會直接談崩。根憑證的「信任鏈」也換了發證機關（Root CA），如果伺服器清單太舊、不認得新的 DigiCert，就不會信任這次連線。

Proxy 代理類型

正向代理（Forward）隱藏客戶端 IP，常見於公司內網翻牆。HTTPS 代理透過 HTTP Tunneling（CONNECT 方法）建立隧道，Proxy 只負責把封包丟過去，不解密內部 TLS 內容——所以 Webhook 握手失敗不是 Proxy 的錯，而是兩端點談不攏。

真相

兇手：HttpClientHandler

犯案手法：未正確將 TLS 版本和憑證驗證設定套用到實際的 HTTP Request

破案關鍵：透過命令列手動測試，發現跳過憑證檢查可成功，進而鎖定 TLS/SSL 層問題。最終發現 HttpClientHandler 的實作缺陷，改用 HttpWebRequest 解決。

啟示：

  • 即使程式碼 3 年沒動，環境變化（憑證過期/更新）也可能導致問題
  • 封測與正式的差異不只是 config，OS/runtime 行為也可能不同
  • 有時候 framework 的抽象層反而會隱藏問題
  • 系統性排查法（逐層排除）是解決未知問題的最佳策略
